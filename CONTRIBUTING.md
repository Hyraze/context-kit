# Contributing to context-kit

## Quick start

```bash
git clone <repo>
cd context-kit
npm install
npm run dev        # http://localhost:5173
npm test           # run all tests
npm run lint       # ESLint
npx tsc --noEmit   # type check
```

Node ≥ 22 required (see `.nvmrc`).

---

## How to add a new tool

Tools live in [src/data/tools.ts](src/data/tools.ts). Each tool is a `Tool` object added to the `TOOLS` array.

### 1. Add the tool object

```ts
{
  id: 'my-tool',           // kebab-case, unique
  name: 'My Tool',         // display name
  category: 'Frontend',    // must be one of CATEGORIES
  emoji: '🔧',
  description: 'One-line description',
  tags: ['keyword1', 'keyword2'],
  versions: ['3', '4'],    // optional — selectable major versions
  envVars: [               // optional — added to .env.example
    'MY_TOOL_API_KEY=',
  ],
  bestPractices: [
    // 3–8 short, actionable rules. These are injected verbatim into every
    // generated config as non-negotiable rules. Be specific, not generic.
    'Always validate inputs with My Tool schemas before processing',
    'Never bypass the My Tool middleware in production routes',
  ],
  adr: {
    title: 'Use My Tool for X',
    context: 'One or two sentences describing the problem this decision solves.',
    decision: 'One sentence describing what was chosen.',
    positives: ['Benefit A', 'Benefit B'],
    negatives: ['Trade-off A'],
    alternatives: ['Alternative X — rejected because...'],
  },
  knowledge: {
    overview: `
## Overview
Two to four paragraphs covering what My Tool is, its core mental model,
and when to reach for it vs. alternatives.
`.trim(),

    patterns: `
## Key Patterns

### Pattern Name
Description of the pattern and a short code example.

\`\`\`ts
// example
\`\`\`
`.trim(),

    gotchas: `
## Common Gotchas

- **Gotcha title**: Explanation of the pitfall and how to avoid it.
- **Another gotcha**: ...
`.trim(),

    quickstart: `
\`\`\`bash
npm install my-tool
\`\`\`

Minimal working example in 5–10 lines.
`.trim(),
  },
}
```

### 2. Optionally add it to a preset

Presets are defined in the `PRESETS` export at the bottom of `tools.ts`. If your tool fits an existing preset, add its `id` to the `tools` array of that preset.

### 3. Add cross-tool combo rules (optional)

If your tool has important integration rules with another tool that's already in context-kit, add a `ToolCombo` to [src/data/combos.ts](src/data/combos.ts):

```ts
{
  tools: ['my-tool', 'other-tool'],
  label: 'My Tool + Other Tool',
  rules: [
    'Use My Tool's adapter for Other Tool — do not write raw queries alongside it',
  ],
  notes: `
### My Tool + Other Tool patterns
Markdown describing integration patterns and gotchas specific to using both together.
`.trim(),
}
```

### 4. Run the tests

```bash
npm test
```

The smoke tests in [src/lib/generator.test.ts](src/lib/generator.test.ts) verify that:

- Every tool has required fields (`id`, `name`, `bestPractices`, `knowledge.overview`)
- All tool IDs are unique
- All preset tool IDs refer to real tools

## How to add a new assistant

Assistants live in [src/data/assistants.ts](src/data/assistants.ts) in the `ASSISTANTS` array. Each assistant generates one or more output files from the selected tools.

### 1. Add the assistant object

```ts
{
  id: 'my-assistant',       // kebab-case, unique
  name: 'My Assistant',
  logo: '🤖',
  badge: 'myassistant.com', // short subtitle shown in the UI
  description: 'One-line description of what files are generated',
  usageSteps: [
    // Shown in the "How to use" section after download.
    // Wrap filenames or commands in backticks for code formatting.
    'Extract the zip into your project root',
    'My Assistant reads `.myassistant-config` automatically',
    'Restart My Assistant to apply the new context',
  ],
  generate(tools, projectName, options) {
    return [
      {
        path: '.myassistant-config',
        content: `# Generated by context-kit
# ${projectName}

${stackRulesSection(tools, options)}
`,
      },
    ]
  },
}
```

### Available builder helpers (already imported in the file)

| Helper | Purpose |
|--------|---------|
| `stackHeader(tools, options)` | Bullet list of tools with optional version |
| `stackRulesSection(tools, options)` | Per-tool rules + combo rules + custom team rules |
| `comboNotesSection(tools)` | Patterns/gotchas for active tool combinations |
| `generatedHeader()` | `[//]: # (Generated by context-kit …)` comment |
| `planCommand(tools)` | Stack-aware `/plan` slash command content |
| `codeReviewCommand(tools)` | Stack-aware `/code-review` slash command content |
| `buildFixCommand(tools)` | Stack-aware `/build-fix` slash command content |
| `generateADRFiles(tools)` | ADR template + index + one ADR per tool |

The `options` object contains `selectedVersions` and `customRules`. Pass it through to `stackHeader` and `stackRulesSection` so versions and team rules appear in the output.

### 2. Run the tests

```bash
npm test
```

The smoke test `every ASSISTANT generates files for a minimal tool set` will run your new assistant against the first tool and verify it produces at least one non-empty file.

## Project structure

```
src/
  data/
    tools.ts        # Tool definitions + PRESETS + CATEGORIES
    assistants.ts   # Assistant definitions + shared content builders
    combos.ts       # Cross-tool combination rules
  components/
    App.tsx         # Wizard shell + URL state
    ToolPicker.tsx  # Step 1 — tool selection
    AssistantPicker.tsx  # Step 2 — assistant selection
    Preview.tsx     # Step 3 — file preview + download
  lib/
    generator.ts    # mergeFiles, generateContextKit, downloadZip
  main.tsx          # React entry point + ErrorBoundary
```

## Code conventions

- TypeScript strict mode — no `any`, no non-null assertions without a comment explaining why
- `noUncheckedIndexedAccess` is on — use `arr[0]!` or optional chaining for array access
- React functional components only — no class components (except `ErrorBoundary` in main.tsx)
- Vitest + React Testing Library for tests — no mocking the DOM, prefer `fireEvent` over `userEvent` for simple interactions
- ESLint + Husky enforce lint on every commit (`npx lint-staged`)
- Conventional commits: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`
